# v0.12.0 迁移指南

从 v0.11.x 迁移到 v0.12.0

---

## 概览

VibeCollab v0.12.0 引入了**YAML-first文档系统**，这是从v0.11.x迁移的最大变化。文档现在以YAML格式存储在`docs/*.yaml`，Markdown文件(`docs/*.md`)是自动生成的视图。

### 主要变更

1. **文档格式**: Markdown → YAML (源文件)
2. **新命令**: `vibecollab docs` 命令组
3. **新命令**: `vibecollab plan` 工作流命令
4. **新功能**: Insight派生链和自动检测
5. **预置工作流**: daily-sync, release-prep, insight-collect

---

## 迁移步骤

### 步骤 1: 升级 VibeCollab

```bash
pip install --upgrade vibe-collab
```

验证版本:

```bash
vibecollab --version
# 应显示 v0.12.0 或更高
```

### 步骤 2: 将现有Markdown文档转换为YAML

v0.12.0 会自动识别并保留现有YAML文件。如果你有旧的Markdown-only文档，需要手动迁移或使用升级脚本。

#### 手动迁移检查清单

对于每个文档文件，检查:

| 文件 | 操作 |
|------|------|
| `docs/CONTEXT.md` | 检查是否有对应的`docs/context.yaml` |
| `docs/DECISIONS.md` | 检查是否有对应的`docs/decisions.yaml` |
| `docs/CHANGELOG.md` | 检查是否有对应的`docs/changelog.yaml` |
| `docs/ROADMAP.md` | 检查是否有对应的`docs/roadmap.yaml` |
| `docs/PRD.md` | 检查是否有对应的`docs/prd.yaml` |
| `docs/QA_TEST_CASES.md` | 检查是否有对应的`docs/qa.yaml` |

### 步骤 3: 生成YAML源文件(如缺失)

如果某些YAML文件缺失，你需要创建它们。以下是模板:

#### context.yaml 模板

```yaml
kind: context
version: "1"
updated_at: "2026-04-02"

status:
  overall: "正常开发中"
  health_score: 85
  active_roles:
    - dev
    - architect
  current_milestone: "v0.12.0"
  completion_percentage: 88

tech_debt:
  - description: "待修复的技术债务"
    priority: medium
    added_at: "2026-04-01"
```

#### decisions.yaml 模板

```yaml
kind: decisions
version: "1"

records:
  - id: "DECISION-001"
    level: "S"
    title: "架构决策示例"
    status: "accepted"
    date: "2026-04-01"
    context: "决策背景"
    decision: "具体决策内容"
    consequences:
      - "影响1"
      - "影响2"
```

#### changelog.yaml 模板

```yaml
kind: changelog
version: "1"

unreleased:
  release_version: "unreleased"
  title: "v0.12.0 (开发中)"
  added:
    - description: "新功能描述"

releases:
  - release_version: "v0.11.0"
    date: "2026-04-01"
    added:
      - description: "之前的变更"
```

#### roadmap.yaml 模板

```yaml
kind: roadmap
version: "1"
updated_at: "2026-04-02"

milestones:
  - id: "v0.12.0"
    title: "YAML-first文档系统"
    status: "in_progress"
    target_date: "2026-04-15"
    items:
      - description: "功能1"
        status: "done"
      - description: "功能2"
        status: "in_progress"
```

#### prd.yaml 模板

```yaml
kind: prd
version: "1"
updated_at: "2026-04-02"

requirements:
  - id: "REQ-001"
    title: "需求标题"
    priority: "high"
    status: "in_progress"
    description: "需求描述"
    acceptance_criteria:
      - "验收标准1"
      - "验收标准2"
```

#### qa.yaml 模板

```yaml
kind: qa
version: "1"
updated_at: "2026-04-02"
case_id_pattern: "TC-{module}-{seq}"

phases:
  - title: "测试阶段1"
    description: "阶段描述"
    cases:
      - id: "TC-MOD-001"
        title: "测试用例标题"
        steps:
          - "步骤1"
          - "步骤2"
        expected:
          - "预期结果1"
        status: "pass"

summary:
  total: 1
  pass: 1
  partial: 0
  fail: 0
  skip: 0
```

### 步骤 4: 渲染Markdown文档

创建或更新YAML文件后，渲染为Markdown:

```bash
# 渲染所有文档
vibecollab docs render --all

# 验证渲染结果
ls docs/*.md
```

### 步骤 5: 更新Git忽略配置(可选但推荐)

由于Markdown文件是自动生成的，你可以选择不将它们纳入版本控制。编辑 `.gitignore`:

```gitignore
# VibeCollab generated files (可选)
# 如果你选择不跟踪生成的Markdown文件
docs/CONTEXT.md
docs/DECISIONS.md
docs/CHANGELOG.md
docs/ROADMAP.md
docs/PRD.md
docs/QA_TEST_CASES.md
```

**注意**: 目前项目仍建议跟踪Markdown文件，以便于GitHub查看。

### 步骤 6: 验证迁移

```bash
# 1. 检查所有YAML文件有效
vibecollab docs validate docs/context.yaml
vibecollab docs validate docs/decisions.yaml
vibecollab docs validate docs/changelog.yaml
vibecollab docs validate docs/roadmap.yaml

# 2. 列出可渲染文档
vibecollab docs list

# 3. 运行合规检查
vibecollab check

# 4. 运行健康检查
vibecollab health
```

## 更新开发工作流

### 旧工作流 (v0.11.x)

```bash
# 编辑Markdown文件
git add docs/CONTEXT.md
git commit -m "更新文档"
```

### 新工作流 (v0.12.0)

```bash
# 编辑YAML文件
vibecollab docs render --all  # 或让它在check时自动渲染
git add docs/*.yaml docs/*.md
git commit -m "更新文档"
```

### 使用 daily-sync 工作流 (推荐)

```bash
# 运行预置的日常同步工作流
vibecollab plan run daily-sync

# 这将自动:
# 1. 检查工作目录
# 2. 验证项目配置
# 3. 运行合规检查
# 4. 渲染所有文档
# 5. 提交并推送更改
```

## 新功能使用

### 1. 文档管理

```bash
# 查看可渲染文档
vibecollab docs list

# 输出示例:
# ┌─────────────────┬─────────────┬──────────────────┬─────────┐
# │ YAML Path       │ Kind        │ Output Name      │ Version │
# ├─────────────────┼─────────────┼──────────────────┼─────────┤
# │ context.yaml    │ context     │ CONTEXT.md       │ 1       │
# │ changelog.yaml  │ changelog   │ CHANGELOG.md     │ 1       │
# │ ...             │ ...         │ ...              │ ...     │
# └─────────────────┴─────────────┴──────────────────┴─────────┘
```

### 2. 工作流自动化

```bash
# 列出可用工作流
vibecollab plan list

# 试运行(不实际执行)
vibecollab plan run daily-sync --dry-run

# 实际运行
vibecollab plan run daily-sync
```

### 3. Insight派生链

```bash
# 查看Insight派生图
vibecollab insight graph --show-derivation

# 创建带自动派生检测的Insight
vibecollab insight derive \
  --title "新的最佳实践" \
  --tags "best-practice,python" \
  --category technique \
  --scenario "处理循环依赖" \
  --approach "使用接口抽象层解耦" \
  --source-task TASK-DEV-001
```

## 常见问题

### Q: Markdown文件被覆盖，我的更改丢失了?

**A**: 从v0.12.0开始，**不要直接编辑Markdown文件**。它们是从YAML自动生成的。请编辑对应的YAML文件，然后运行 `vibecollab docs render --all`。

### Q: 我可以在YAML中使用Markdown格式吗?

**A**: 可以! YAML文件的字符串值支持Markdown格式。渲染时会保留格式。

### Q: 如何添加自定义文档类型?

**A**: v0.12.0支持6种标准类型。自定义类型需要修改模板引擎，将在未来版本中支持。

### Q: 现有的plan run命令还能用吗?

**A**: 可以。v0.12.0的 `plan run` 向后兼容，支持任意YAML计划文件。新增的 `plan list` 和预置工作流是额外的便利功能。

### Q: 如何禁用自动文档渲染?

**A**: 目前没有自动渲染。你需要手动运行 `vibecollab docs render --all`。可以配置Git钩子来自动执行:

```bash
vibecollab hooks install
# 编辑 .git/hooks/pre-commit 添加:
# vibecollab docs render --all
```

## 回滚到v0.11.x

如果需要回滚:

```bash
pip install "vibe-collab<0.12.0"
```

你的YAML文件仍然会被识别，但v0.11.x不会自动渲染它们。你需要手动维护Markdown文件的一致性。
